IAM App: http server bootstraping (practical activity)

Learning outcomes

  • Build a simple HTTP server using fastify.
  • Validate inputs and serialise payloads using JSON schemas.
  • Generate TypeScript interfaces from JSON schemas.
  • Work with promises, async error handling, error serialisation, etc.
  • Write specs, implement entity factories, compute a test coverage report.
  • Implement session management on stateless workers.
  • Ensure database integrity with transactions.
  • Set up a simple resource-based and action-based policy framework.

Specifications

In this tutorial, you will set up an HTTP server and the necessary endpoints for registration and login. If not already done, I suggest you check the associated presentation called HTTP protocols, servers & libraries.

The only new libraries allowed for installations are: fastify, @fastify/cookie, cookie-signature, ajv (also a dependency of fastify) and the corresponding @types/* packages.

No UI is expected: you have to implement and test your endpoints mainly with automated integration tests. All payloads and responses are expected to be JSON-encoded.

REST endpoint naming conventions

This tutorial series follows REST naming convention for as many endpoints as possible. The REST naming convention is about how to map CRUD operations on resources to the corresponding HTTP verbs and HTTP paths.

A word about JSON-schemas

JSON schema is a declarative language specifying the expected shape of a JSON payload. It is primarily used to validate the schema compliance of a payload before consuming it (as done by the ajv package). It could also be leveraged for documentation purposes (ex. openapi documentation). Some libraries even leverage it for entity serialisation (ex. fast-json-stringify). Fastify already depends on ajv and fast-json-stringify, so you not need to install them manually.

That said, JSON schemas are a good candidate for input validation and safe output serialisation, with near zero-cost documentation included. While DTO paradigm is very popular in Java-like world (and still suitable for js), JSON schema can also be a good replacement/complement fit. Fastify uses them for its core Validation and Serialisation feature, as we will also do. In addition, the fastify implementation is built for high performance and plays nicely with typescript.

Exercise 1: write three json schemas that match the following typescript interfaces. Be sure the type of each property exactly matches (including requireness) and that no other property is given. Please write the json schemas yourself and do not use automated tools: the goal is to get familiar with the syntax and concepts.

For now, you can store the json files inside a dedicated folder of your repository. To enable json schema-related tooling in your IDE (like autocomplete), start by adding a $schema property (see online examples).

interface CreateUserResponseBody {

Bonus challenge 1: json schema utils (hard)

This optional challenge is reserved for advanced typescript developers. Consider other exercises first and come back once your job is done.

In the end, the application will use json schema for http boundaries (in and out). Considering that the schemas of a REST API endpoints result a lot of duplicated code, you will think about various utilities. Those utilities are intended to build, compose partial json schemas together.

The following code snippet illustrates the use of those helpers. You have to reverse engineering mergeSchema, pickFromSchema and propertiesToObjectSchema.

Pay attention to the way typescript is able to infer a static type from a JSON schema (ex. UserCreateRequestBody type is inferred as an object having email and passwordConfirmation properties, among others). To operate property, type definitions of the utilities methods should preserve as much information as possible (ex. property names, types, requireness…) during transformations.

import { JTDDataType } from 'ajv/dist/jtd'
import { mergeSchema, pickFromSchema, propertiesToObjectSchema } from './schema-utils'

export const userSchema = propertiesToObjectSchema({
  id: { type: 'string' },
  email: { type: 'string' },
  firstname: { type: 'string' },
  lastname: { type: 'string' }
} as const)

export const userCreateRequestBody = mergeSchema(
  pickFromSchema(userSchema, 'firstname', 'lastname', 'email'),
  propertiesToObjectSchema({ password: { type: 'string' }, passwordConfirmation: { type: 'string' } } as const)
)

export const userCreateResponseBody = pickFromSchema(userSchema, 'id', 'firstname', 'lastname', 'email')

export type UserCreateRequestBody = JTDDataType<typeof userCreateRequestBody>
export type UserCreateResponseBody = JTDDataType<typeof userCreateResponseBody>

Langage du code : TypeScript (typescript)

By design, Typescript only checks types at the build time, but asserts nothing at runtime. Supposing that the static types follow good practices (ex. banning the any type), it gives you confidence the runtime types will match.

But once your application starts consuming inputs (ex. a JSON payload sent to a HTTP endpoint), you need both static definitions and runtime assertions. That could become a challenge to keep the static and runtime types consistent together. Whenever it is possible, I am trying to derive one from the other, so I keep a single source of truth.

Exercise 3: infer the static type of input/outputs from the corresponding json schemas. Fastify has a guide on that topic, but every mentioned solution requires the installation of a new library (ex. typebox, json-schema-to-typescript, etc.). On another side, I recently discovered that ajv can also fit the need. Pick one solution at your choice, be ready to discuss the pros and cons.

Enforce a response.schema to always be set

When dealing with fastify’s responses, you are encouraged to return a promise resolving in a serializable payload. That payload then passes through the json schema derived serializer.

Look at the following poorly implemented snippet. It has 3 main flaws:

fastify.get<{ Params: { id: string } }>(
  '/:id',
  req => dataSource.getRepository(User).findOneBy({ id: req.params.id })
)
Langage du code : TypeScript (typescript)
  • First, the route lacks runtime validations for the params. It could be reasonable to assume that id is always a string since it is parsed from the path. But adding a json schema (supposing the static type is derived from the json schema) will prevent most typo errors. AJV can also coerce the type based on JSON schema (ex. to cast the id as number if it was supposed to be).
  • The second issue is more dramatic. There isn’t any policy control in place. The client can submit any id and fetch associated account information, leading to personal data disclosure. Let’s come back to this later.
  • Last, but not least, the code returns Promise<User> without any response schema being defined. That basically tells fastify to stringify all enumerable properties (including passwordHash) and send them over the network! Let’s focus on this first.

To ensure no one never forgets the response serializer, you will add a runtime assertion at startup. By leveraging the hooks API, you can register a callback triggered every time a route is mounted onto the fastify instance. The hook you need is called onRoute. Please refer to the official documentation of other available hooks.

What we are doing now is called « security by design », because it becomes impossible to meet the risky circonstances anywere in our application. When possible and relevant, I recommand you solve issues (including non-security related one) at the application level. Why? Think about it yourself!

Your code will probably look similar to this:

// src/lib/fastify.ts
import fastify, { RouteOptions } from 'fastify'
import { webApiRoutes } from '../routes/web-api/web-api-routes'
import { FASTIFY_LOGGING } from './dotenv'

export const server = fastify({ logger: FASTIFY_LOGGING })
  .addHook('onRoute', assertsResponseSchemaPresenceHook)
  .register(webApiRoutes, { prefix: '/web-api' })

export function assertsResponseSchemaPresenceHook (routeOptions: RouteOptions) {
  // do it yourself

Exercise 5: update the default ajv configuration to set removeAdditional: false. This is the default behaviour of avj that was overridden by fastify. This way, any unknown additional property always throws an error instead of being silently dismissed. Read the manual.

When dealing with a large number of attributes per model, mass assignment paradigm is pretty convenient. But it also causes security weaknesses if assigned attributes are not carefully whitelisted (even Github got caught).

Example of mass assignment using the merge method from typeorm.

const repo = dataSource.getRepository(User)
const user = await repo.findOneBy({ id: req.params.id })
repo.merge(user, req.body.user)
await repo.save(user)

Exercise 6: for the same reason as we enforced a response schema to always be set, we will change the default fastify behaviour when body, query or params schema is missing. In the absence of a validation schema, just drop the input. Because silently dropping the input degrades the debugging experience, ensure the application responds with a 400-status code and a meaningful message. It is possible (although not mandatory) to set up this behaviour using the onRoute hook. TDD it.

Bonus: futher reading about a mass-assignment related vulnerability that operates with Object.assign() and JSON.parse(), leading a prototype polution (remote code execution, aka. RCE).

Safer default error handler and serializer

Let’s take care of the security during error handling, as no sensitive information should leak. One of the attack vectors is error serialisation when the app sends internal errors messages.

For 4xx related errors, it is generally acceptable to send the failure’s reason to the user. This way, the client (that could be developers in case of an API) has the opportunity to retry with fixed parameters.

400 Bad Request

The HyperText Transfer Protocol (HTTP) 400 Bad Request response status code indicates that the server cannot or will not process the request due to something that is perceived to be a client error (for example, malformed request syntax, invalid request message framing, or deceptive request routing).

Mozilla Developer Network

403 Forbidden

The HTTP 403 Forbidden response status code indicates that the server understands the request but refuses to authorize it.

Mozilla Developer Network

404 Not Found

The HTTP 404 Not Found response status code indicates that the server cannot find the requested resource. Links that lead to a 404 page are often called broken or dead links and can be subject to link rot.

Mozilla Developer Network

That is why the following response is acceptable:

{
    "statusCode": 400,
    "error": "Bad Request",
    "message": "body should have required property 'hello'"
}
